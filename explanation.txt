CERTIFICATE GENERATOR v2.0 — CODE EXPLANATION
=============================================

This document explains the architecture, design patterns, and key code
sections of the Certificate Generator project for CSC 226.


1. OVERALL ARCHITECTURE
----------------------

The project follows a layered design:

  main.cpp  (entry point)
     |
     v
  HttpServer  (web server, accepts HTTP connections)
     |
     v
  CertificateGenerator  (data manager, CRUD, search, sort, stats)
     |
     v
  Certificate <── CertificateTemplate  (polymorphic certificate + strategy)
     |
     v
  Person → Student  (inheritance hierarchy)

Data flow: Browser → HTTP (JSON) → HttpServer → CertificateGenerator
→ Certificate + Template → HTML file on disk.


2. FILE-BY-FILE BREAKDOWN
-------------------------

a) include/exceptions.hpp  —  Custom exception classes
   ├─ ValidationException — thrown when student data fails validation
   └─ FileException — thrown when a file cannot be opened
   Both inherit from std::exception and override what().

b) include/person.hpp + src/person.cpp  —  Inheritance hierarchy
   ├─ Person (abstract base)
   │   └─ pure virtual getDescription() — every subclass must define this
   ├─ Student (concrete subclass)
   │   ├─ gradeToPoints() — converts "A", "B+", or "93" to 4.0 GPA scale
   │   └─ isValidGrade() — static, validates before storage
   └─ Demonstrates: inheritance, polymorphism, pure virtual functions.

c) include/certificate.hpp + src/certificate.cpp  —  Two design patterns
   ├─ STRATEGY PATTERN: CertificateTemplate (interface)
   │   ├─ FormalTemplate — double border, serif font
   │   ├─ ModernTemplate — gradient background, rounded corners
   │   └─ MinimalistTemplate — monospace, clean lines
   │   The visual style can be swapped at runtime without changing
   │   the certificate logic.
   ├─ TEMPLATE METHOD PATTERN: Certificate (abstract)
   │   ├─ generateHTML() calls: header() → styles() → body()
   │   ├─ CertificateOfExcellence
   │   ├─ CertificateOfCompletion
   │   └─ CertificateOfParticipation
   │   The algorithm skeleton is fixed; subclasses only provide
   │   the type-specific strings.
   └─ FACTORY PATTERN: makeTemplate() and makeCertificate()
       These functions return the correct subclass based on an enum.

d) include/generator.hpp + src/generator.cpp  —  Central data manager
   ├─ std::map<std::string, Student> — database keyed by name
   ├─ CRUD: addStudent, removeStudent, findStudent (case-insensitive)
   ├─ Search: by name, course, or GPA range
   ├─ Sort: by name, grade, or course (using std::sort with lambdas)
   ├─ CSV parser with per-line error handling
   ├─ Statistics: per-course counts, grade distribution, average GPA
   └─ Batch/single certificate generation via factory delegation

e) include/http_server.hpp + src/http_server.cpp  —  Web UI server
   ├─ Uses Winsock2 to listen on localhost:8080
   ├─ Serves embedded HTML/CSS/JS (single-page app, ~400 lines)
   ├─ REST-like API endpoints:
   │   GET  /          → serves the web page
   │   GET  /list      → JSON array of all students
   │   GET  /search?q= → filtered JSON results
   │   GET  /stats     → JSON statistics object
   │   POST /add       → insert one student
   │   POST /remove    → delete one student
   │   POST /load      → batch import from CSV
   │   POST /generate  → create certificate(s)
   │   POST /sort      → reorder the student list
   ├─ Each request runs in its own std::thread (detached)
   ├─ Auto-opens the browser via ShellExecuteA
   └─ The frontend uses fetch() to call back to C++ — no page reloads

f) main.cpp  —  Entry point
   Creates a CertificateGenerator, wraps it in an HttpServer on port
   8080, and starts the accept loop. Blocks until process is killed.

g) CMakeLists.txt  —  Build system
   ├─ C++17 standard required
   ├─ Two targets: certificate_generator (main) and test_certificate
   └─ Links ws2_32 for Winsock support

4. KEY CODE WALKTHROUGH
-----------------------

a) Polymorphic certificate generation (certificate.cpp:196-204):

    std::string Certificate::generateHTML() const {
        std::string typeName = getTypeName();         // virtual call
        std::string html;
        html += template_.header(typeName);            // strategy call
        html += template_.styles();                    // strategy call
        html += template_.body(student_, typeName);    // strategy call
        html += "</body>\n</html>";
        return html;
    }

    This is the Template Method pattern. The algorithm is fixed in
    the base class. The getTypeName() call uses dynamic dispatch to
    get "Excellence", "Completion", or "Participation". The three
    template_ calls use dynamic dispatch to select the visual style.

b) Case-insensitive student lookup (generator.cpp:37-42):

    Student* CertificateGenerator::findStudent(
        const std::string& name)
    {
        std::string lower = toLower(name);
        auto it = std::find_if(students_.begin(), students_.end(),
            [&](const auto& pair) {
                return toLower(pair.first) == lower;
            });
        return (it != students_.end()) ? &it->second : nullptr;
    }

    Uses a lambda with std::find_if to do a case-insensitive linear
    scan of the map. The toLower() function converts both the query
    and each key before comparing.

c) CSV parsing with error tolerance (generator.cpp:135-162):

    while (std::getline(file, line)) {
        lineNum++;
        if (line.empty()) continue;
        // ... parse CSV fields ...
        try {
            addStudent(Student(name, course, grade));
            loaded++;
        } catch (const ValidationException& e) {
            std::cerr << "  [Line " << lineNum
                      << " skipped: " << e.what() << std::endl;
        }
    }

    Each line is processed independently. If one line fails validation
    (e.g. empty grade), it is reported to stderr and processing
    continues with the next line. No data is lost.

d) HTTP server routing (http_server.cpp:500+):

    The handleClient function parses the raw HTTP request, extracts
    the method and path, and dispatches to the correct handler using
    if/else chains. Each handler calls the appropriate
    CertificateGenerator method and returns JSON.

    For example, GET /list calls gen.getAllStudents() and serialises
    the result as a JSON array. POST /add calls parseForm(),
    creates a Student, and calls gen.addStudent().

e) Browser auto-open (http_server.cpp:680):

    ShellExecuteA(nullptr, "open", url.c_str(), nullptr,
                  nullptr, SW_SHOWNORMAL);

    This Windows API call opens the default browser to the server
    URL so the user doesn't have to type it manually.


5. BUILD & RUN
--------------

  CMake:
    cmake -S . -B build
    cmake --build build
    ./build/certificate_generator

  Direct g++:
    g++ -std=c++17 -I include -o certificate_generator.exe \\
        main.cpp src/*.cpp -lws2_32
    ./certificate_generator

  The server starts on http://localhost:8080 and the browser
  opens automatically.
